# ADR-008: Invite Link Storage and Auth — Firestore

**Status:** Decided  
**Date:** 2026-05-30

## Context

The product pivot introduces a new auth model: visitors don't authenticate with Google. Instead, they present a UUID invite link generated by Christian. This requires:

1. A place to store invite documents: `{uuid, created_at, label, status}`
2. A mechanism to validate links at session start (is it valid? is it expired?)
3. A mechanism to generate links from GitHub Actions
4. Optional: automatic cleanup of expired links

Constraints:
- Visitor-facing: must be fast to read (link validation happens on every WebSocket connect)
- Admin-side: link generation happens sporadically (a few per week)
- Scale: negligible (dozens of documents at most)
- GCP-native preferred (same billing, same service account)

Alternatives evaluated:
- **GCS JSON file:** Read-modify-write a single JSON file. Not atomic — concurrent writes (two visitors opening sessions simultaneously) can corrupt the file. No TTL. Rejected.
- **SQLite on Cloud Run:** Resets on every new Cloud Run revision deploy (ephemeral filesystem). Persistent volumes are possible but add ops complexity. Rejected.
- **Cloud SQL (Postgres):** Full relational DB at ~$7/month minimum. Overkill. Rejected.
- **GitHub Secrets:** Write-only from GitHub Actions perspective; not readable by the backend at runtime. Rejected.
- **Firestore:** Document DB, GCP-native, serverless, free tier, atomic writes, TTL-based auto-expiry, readable from Cloud Run via service account. Selected.

## Decision

Use **Firestore** (Native mode) for invite link storage. One collection: `invite_links`. Each document keyed by UUID.

**Document schema:**
```
invite_links/{uuid}
  created_at: Timestamp       ← Firestore server timestamp
  label: string               ← Christian's reference note ("Meeting with Max")
  status: "active" | "revoked"
  ttl: Timestamp              ← created_at + 10 days; used by Firestore TTL policy for auto-delete
```

**Validation logic (backend, on every WebSocket connect):**
```python
async def validate_invite(uuid: str) -> bool:
    doc = await db.collection("invite_links").document(uuid).get()
    if not doc.exists:
        return False
    data = doc.to_dict()
    if data["status"] != "active":
        return False
    age = datetime.utcnow() - data["created_at"].replace(tzinfo=None)
    if age.days >= 10:
        return False
    return True
```

**Link generation (GitHub Actions):**
```python
# generate_invite.py — run by GitHub Actions workflow
import uuid, os
from google.cloud import firestore
from datetime import datetime, timedelta

db = firestore.Client()
invite_id = str(uuid.uuid4())
label = os.environ.get("INPUT_LABEL", "")
now = datetime.utcnow()
db.collection("invite_links").document(invite_id).set({
    "created_at": now,
    "label": label,
    "status": "active",
    "ttl": now + timedelta(days=10),
})
base_url = os.environ["APP_BASE_URL"]
print(f"Invite URL: {base_url}/?invite={invite_id}")
```

## Rationale

- **Free at personal scale:** Firestore's free tier (1GB storage, 50K reads/day, 20K writes/day) will never be hit with a handful of invite links per month.
- **Atomic writes:** No race conditions on link validation or creation.
- **TTL auto-expiry:** Firestore supports a TTL policy on a field. Set `ttl` to `created_at + 10 days`; Firestore deletes the document automatically. No cron job or cleanup lambda needed.
- **GCP-native auth:** Cloud Run's service account can be granted `roles/datastore.user` via Terraform. No additional credentials.
- **GitHub Actions access:** The GitHub Actions workflow uses Workload Identity Federation to authenticate, then calls `google.cloud.firestore` in `generate_invite.py`.

## What This Option Does NOT Do Well

- Firestore TTL deletion is best-effort (may take up to 24h after expiry). The backend must NOT rely on document absence as the validity check — it must check `created_at + 10 days` explicitly. TTL is for storage cleanup only.
- Firestore is a NoSQL document store. Querying by `created_at` range (e.g., "show all active links") requires a composite index. For MVP (Christian looking at GitHub Actions log for links), this is irrelevant.
- If Christian accidentally deletes the Firestore database, all active invite links are lost. There's no recovery path. Acceptable at this scale.

## Consequences

- Terraform must provision: Firestore database (Native mode, `europe-west1`), Firestore TTL policy on the `ttl` field, IAM binding for the Cloud Run service account (`roles/datastore.user`), IAM binding for the GitHub Actions service account.
- GitHub Actions `generate_invite.yml` workflow: `workflow_dispatch` with optional `label` input, runs `generate_invite.py`, outputs URL to job log.
- PWA: Extracts `?invite=` from URL on load. If missing: show static "invalid link" page. If present: WebSocket connect includes it for server-side validation.
- Backend: Validates UUID on every new WebSocket connection before opening a Gemini Live session. Invalid/expired → close WebSocket with code 4001, PWA shows error message.
